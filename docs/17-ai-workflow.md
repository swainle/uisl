# AI 工作流

UISL 的 AI 工作流分为两个独立阶段：先把草稿和需求整理为 `.sl`，再根据 `.sl` 与 Profile 生成或更新目标代码。两个阶段的输入、目标和约束不同，不得混为一次自由发挥式生成。

```text
草稿 / 截图 / HTML / PRD / 说明
        ↓
AI：遵循 UISL 标准生成或更新 `.sl`
        ↓
UISL Core + Profile
        ↓
AI：读取目标模块并强制遵循 Ponytail
        ↓
目标平台代码
```

## 一、模块范围

AI 不默认读取整个大型仓库，只读取当前模块及其真实依赖。

### Profile 已存在

以 `profile.outputRef` 指向的实现目录为目标模块。

### Profile 尚未存在

以用户指定的实现目录为目标模块。生成 Profile 时补充 `profile.outputRef`。

### 必须读取

```text
UISL：
- 当前模块相关的全部 `.sl`
- 当前 `.sl` 引用的 layout、component、design、token、profile 和其他文件

目标代码：
- 目标模块目录下的全部代码和配置
- 目标模块 package.json、tsconfig 和框架配置
- 仓库根 workspace 配置
- package-lock.json
- 目标模块实际 import 的共享代码
```

### 不默认读取

与当前模块没有引用关系、没有共享依赖、也不属于当前生成批次的其他业务模块。

## 二、阶段一：草稿生成 `.sl`

### 输入

- 用户提供的草稿、截图、HTML、PRD 或说明。
- UISL 标准。
- 当前模块已有的全部 `.sl`。
- 当前模块中与目标 UI 相关的现有代码。
- 用户指定的平台、语言、框架、组件库、样式方案和目标环境。

### 目标

完整表达已经明确的 UI 结构、内容、显示语义、行为、数据、状态、权限、校验、可访问性、响应式和目标实现适配。

### 编写阶梯

新增一条 UISL 声明前，依次判断：

1. 草稿或需求是否明确表达了该语义？未明确且无法从现有实现确认时，不推测。
2. 当前模块已有 `.sl`、layout、component、token、type 或稳定 key 是否可以复用？
3. UISL 现有层和 type 是否已经可以表达？
4. 该内容是否只是目标平台差异？平台差异写入 Profile，不写入 Core。
5. 是否在重复声明可通过组件实例 key 或引用表达的内容？
6. 最后才增加新的 type、state、behavior、token 或 override。

### 强制要求

- Core 保持平台、框架和组件库无关。
- Profile 记录用户指定或默认采用的目标技术栈。
- 不为“未来可能需要”增加状态、行为、抽象或扩展点。
- 不因追求简短而遗漏权限、校验、可访问性、响应式、错误状态或数据语义。
- 信息不足会影响语义正确性时，向用户询问；可以从现有模块明确推导时，直接采用现有模式。

## 三、阶段二：`.sl` 生成目标代码

阶段二必须同时读取：

- 当前 Profile。
- 当前模块相关的全部 `.sl` 及其引用。
- 目标模块全部代码和配置。
- `docs/18-ponytail-code-generation.md`。

先理解当前模块的真实结构、依赖、组件模式、数据流和调用关系，再决定生成哪些文件、组件和逻辑。不得仅根据单个页面 `.sl` 孤立生成代码。

## 四、Profile 默认值

用户指定值优先；未指定时采用以下默认值：

| 目标 | 默认组合 |
|---|---|
| Web React | Next.js + TypeScript + shadcn + Tailwind |
| Web Vue | Vue + Vite + TypeScript + Ant Design Vue |
| 小程序 | Taro + React + TypeScript + NutUI + 微信小程序 |
| 桌面端 | Tauri + Vite + React + TypeScript + shadcn + Tailwind |
| 桌面端 Vue | Tauri + Vite + Vue + TypeScript + Ant Design Vue |
| 包管理器 | npm |

Profile 字段与模板见 `docs/19-default-profiles.md` 和 `templates/profile-*.sl`。

## 五、安装规则

### 允许自动安装

当 Profile 已明确指定，但目标模块缺少以下内容时，使用官方 CLI 或 npm 命令安装：

- 项目或框架骨架。
- Profile 指定的组件库。
- Profile 指定且当前页面实际使用的组件。
- Profile 指定的样式、图标、图表或桌面插件。

### 禁止行为

- 不手写仿制框架、组件库或第三方组件源码。
- Profile 未指定的新第三方依赖不得自行安装。
- 不因为安装不方便而生成临时替代组件。

### 必须停止并报告

- 官方 CLI 或 npm 安装失败。
- 现有模块技术栈与 Profile 冲突。
- 初始化命令可能覆盖现有代码或配置。
- 依赖版本要求无法与现有项目兼容。

不得自动迁移技术栈、清空目录、覆盖已有项目或使用破坏性命令绕过冲突。

## 六、组件边界

组件是否抽取不以固定复用次数决定。AI 必须结合当前生成批次的全部页面和目标模块现有代码综合判断。

一次使用也可以抽取组件，例如：

- 具有稳定业务语义。
- 封装非平凡行为、权限、校验或可访问性。
- Profile 或 `platformAdapter` 要求独立替换。
- 当前批次多个页面可以共享。
- 符合现有项目明确的组件边界。
- 需要独立验证或维护。

不得仅因 JSX/模板较长、为了目录整齐或推测未来复用而抽象；也不得仅因当前只使用一次而拒绝合理抽取。

## 七、更新工作流

后续修改优先使用 `.sl diff`、现有目标代码和 trace 执行最小范围更新：

```text
1. 读取当前模块全部相关 `.sl`。
2. 识别 `.sl` 变化路径。
3. 通过 trace manifest、data-uisl、注释、debug key 或现有符号定位目标代码。
4. 只修改受影响文件和真实共享实现。
5. 保留未受影响的人工修改。
6. 运行当前模块已有验证命令。
```

如果共享根因位于公共组件或函数，应在共享位置修复一次，不在每个页面重复补丁。

## 八、完成报告

AI 完成任务后应简要说明：

- 读取和修改的模块范围。
- 复用了哪些已有实现。
- 新增或安装了哪些 Profile 已授权内容。
- 运行了哪些验证。
- 是否存在阻塞、冲突或明确保留的简化。
