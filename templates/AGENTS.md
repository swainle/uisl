# UISL 模块 AI 执行规则

本文件应复制到每个 UISL 实现模块根目录。请根据项目实际位置调整下方 `docs/uisl` 路径。

## 必须读取

处理草稿生成 `.sl`、根据 `.sl` 生成代码或根据 `.sl` 更新代码前，必须读取：

- `docs/uisl/docs/17-ai-workflow.md`
- `docs/uisl/docs/18-ponytail-code-generation.md`
- 当前 Profile
- 当前模块相关的全部 `.sl` 及其引用
- 当前实现模块的全部代码与配置
- 仓库根 workspace 配置和 `package-lock.json`
- 当前模块实际 import 的共享代码

## 模块边界

- Profile 已存在时，以 `profile.outputRef` 指向目录为实现模块。
- Profile 尚未建立时，以用户指定目录为实现模块。
- 不读取或修改与当前模块无引用关系的其他业务模块。

## 强制要求

- UISL Core 保持平台无关，平台和框架差异写入 Profile。
- 根据 `.sl` 生成或更新代码时，强制遵循 Ponytail 规范。
- 先复用当前模块已有实现，再使用平台、框架和 Profile 声明组件库的原生能力。
- 不创建推测性抽象、无意义 wrapper、未请求的通用运行时或未来脚手架。
- 组件边界根据全部相关页面、语义内聚、已有项目模式、独立行为和平台替换需求判断，不按固定使用次数判断。
- Profile 明确指定但缺失的项目、框架、组件库或组件，通过官方 CLI 或 npm 命令安装；不手写仿制实现。
- Profile 未指定的新依赖不得自行安装。
- 安装失败、技术栈冲突或命令可能覆盖现有文件时停止并报告。
- 权限、校验、可访问性、破坏性操作确认、API 错误处理、安全边界、数据丢失保护和 trace 不得精简。
